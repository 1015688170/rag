# RAG 系统流程说明（甲方版）

## 1. 文档目的

本文档用于说明当前 RAG（Retrieval-Augmented Generation，检索增强生成）系统的整体处理流程、模块职责和业务价值，重点说明系统如何面向运维新人、实习生和一线值班同学，将较为白话的问题转换为可检索、可回答、可追溯的工程问答。

## 2. 系统定位

本系统是一套面向企业内部运维知识库的问答测试台。它不是让大模型直接凭通用知识回答问题，而是先从企业知识库中检索相关资料，再由大模型基于资料生成答案。

系统主要服务以下场景：

- 运维新人不熟悉标准术语，只能用白话描述问题。
- 实习生需要快速理解告警、日志、命令和 SOP。
- 一线值班同学需要查询可执行步骤、配置示例和排障依据。
- 管理或验收人员需要看到答案来源，确认回答是否有据可查。

## 3. 总体架构

系统采用前后端分离架构：

- 前端：基于 React + Vite，负责用户提问、参数配置、提示词调整、答案和来源展示。
- 后端：基于 FastAPI，负责执行完整的 RAG 流程。
- 检索层：基于 Azure AI Search，同时执行关键词检索和向量检索。
- 重排层：基于本地 BGE Reranker，对召回片段做相关性排序。
- 生成层：支持 GPT-4o 与 Claude Opus 4.5 等大模型生成最终回答。

整体链路可以概括为：

用户问题 -> 白话理解与标签化 -> 向量化 -> 混合检索 -> 本地重排 -> 大模型生成 -> 答案与来源返回

## 4. 面向新人白话问题的提示词策略

运维新人和实习生提问时，常见表达往往不够标准。例如：

- “pod 炸了咋看？”
- “CPU 很高怎么查？”
- “流水线挂了，提示 exit code 1 是啥？”
- “Grafana 上想看生产环境容器 CPU，怎么写 PromQL？”

这些问题可能缺少完整的工程术语。为提高回答质量，系统提示词要求大模型先做“问题理解与关键词标签”，再回答问题。

### 4.1 标签体系

系统会将用户问题拆解为以下标签：

- 场景：例如 Kubernetes、GitLab CI、Grafana、Prometheus、镜像扫描、发布变更。
- 对象：例如 Pod、容器、节点、Pipeline、镜像、服务、命名空间。
- 症状：例如 OOM、CPU 高、水位高、Exit Code 1、启动失败、漏洞阻断。
- 指标/命令：例如 PromQL、kubectl、journalctl、rate、container_cpu_usage_seconds_total。
- 意图：例如排查原因、确认依据、生成查询语句、给出 SOP、解释告警。

如果用户没有明确表达某类信息，系统会标记为“未明确”，避免自行脑补。

### 4.2 标签化示例

用户问题：

“生产 pod CPU 很高，Grafana 里咋查？”

系统理解为：

- 场景：Grafana、Prometheus、Kubernetes、生产环境
- 对象：Pod、容器
- 症状：CPU 使用率高
- 指标/命令：PromQL、container_cpu_usage_seconds_total、rate
- 意图：生成查询语句、辅助排查

这种标签化方式的价值是：既照顾新人白话表达，又把问题转成后续检索和回答更容易理解的工程语义。

## 5. RAG 处理流程

用户在前端输入问题后，系统按以下步骤执行：

1. 前端将用户问题、Embedding 模型、聊天模型、召回数量、重排数量、提示词模板等参数提交到后端。
2. 后端根据所选 Embedding 模型，将用户问题转换为向量表示。
3. 后端将原始问题文本和问题向量一起提交给 Azure AI Search，执行混合检索。
4. Azure AI Search 返回候选知识片段。
5. 后端将候选片段交给本地 BGE Reranker，按问题相关性重新排序。
6. 后端选取重排后的前 N 条片段，拼接成大模型参考资料。
7. 后端将系统提示词、参考资料和用户问题提交给选定的大模型。
8. 大模型基于参考资料生成结构化回答。
9. 后端返回答案、来源片段、模型信息和分数信息。
10. 前端展示答案和参考数据源，方便用户核对依据。

## 6. 关键阶段说明

### 6.1 问题输入

用户可以直接用自然语言提问，不要求使用严格术语。系统提示词会要求大模型先识别场景、对象、症状、指标和意图。

### 6.2 向量化

系统将用户问题转换为向量，用于语义检索。向量化可以帮助系统找到“意思相近但文字不完全相同”的知识内容。

当前支持：

- `ada-002`
- `google-005`

### 6.3 混合检索

系统使用 Azure AI Search 执行混合检索：

- 关键词检索：适合命中明确术语、命令、报错码、指标名。
- 向量检索：适合处理白话问题和语义相近表达。

混合检索同时兼顾“字面命中”和“语义理解”，适合新人提问不标准的场景。

### 6.4 本地重排

召回结果可能包含多条片段，其中有些只是弱相关。系统使用 BGE Reranker 对“问题 + 片段”重新打分，挑出最适合送入大模型的内容。

重排的作用是：

- 提高参考资料相关性。
- 减少无关片段干扰。
- 提升最终回答准确性。

### 6.5 答案生成

大模型不会直接自由发挥，而是基于系统提供的参考资料回答。当前提示词要求回答优先采用以下结构：

## 问题标签

展示场景、对象、症状、指标/命令和意图。

## 结论

先给新人能看懂的直接结论。

## 操作步骤

给出可执行的排查步骤、命令或查询语句。

## 注意事项

说明风险、边界和容易误解的地方。

## 参考来源

列出答案依据来自哪些知识片段或文件。

### 6.6 来源返回

系统会将最终使用的参考片段返回给前端，包括：

- 来源文件路径
- 片段预览
- 召回分数
- 重排分数
- 分数来源

这使得甲方可以核验答案依据，降低“模型编造”的风险。

## 7. 提示词控制原则

当前提示词遵循以下原则：

- 面向新人：解释要清晰，避免只给高级术语。
- 先标签化：把白话问题转换为标准运维标签。
- 先结论后步骤：方便值班同学快速执行。
- 强制代码块：命令、YAML、日志、PromQL 必须用 Markdown 代码块展示。
- 严格依据资料：没有资料支撑时明确说明“当前资料中没有足够依据”。
- 保留来源：关键结论、命令、参数和 SOP 步骤尽量标注来源文件。

## 8. 异常与兜底

### 8.1 未检索到有效资料

如果知识库中没有足够依据，系统不会强行生成答案，而是提示资料不足，并说明缺少哪类资料。

### 8.2 重排不可用

如果本地重排模块因依赖、模型文件或运行环境问题异常，系统会自动降级为直接使用召回结果继续回答。

此时前端会显示：

- `score_source = recall`
- `rerank_score` 为空
- 页面提示“重排不可用”

该机制保证系统在局部模块异常时仍具备基本问答能力。

## 9. 系统价值

从业务角度看，本系统具备以下价值：

- 降低新人使用门槛：允许用户用白话描述问题。
- 提升知识利用率：把内部文档、规范和 SOP 转成可问答能力。
- 增强答案可信度：答案附带来源，便于审阅和验收。
- 提升排障效率：将检索、筛选、整理步骤自动化。
- 支持持续优化：提示词、模型、召回数量和重排数量均可调整。

## 10. 总结

本系统的核心不是让大模型直接回答，而是围绕企业知识库建立“白话问题理解 -> 关键词标签化 -> 知识检索 -> 结果重排 -> 基于依据生成答案”的闭环。

该流程既能照顾运维新人和实习生的自然表达习惯，又能保证回答尽量有依据、可追溯、可解释，适合作为企业内部运维知识问答和 RAG 效果验证平台。
